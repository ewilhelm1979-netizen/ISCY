# ISCY V23.7.27

ISCY V23.7.27 focuses on security boundaries, release discipline, and public open-source readiness.

ISCY remains a self-hosted, local-first cybersecurity governance platform under `AGPL-3.0-only`. Regulatory support does not constitute legal advice, certification, conformity assessment, or an audit opinion.

## Security hardening

- Evidence files are no longer exposed through a public static `/media/` path.
- Authenticated Evidence downloads are tenant-scoped and protection-class aware.
- Missing and foreign-tenant Evidence identifiers use the same not-found behavior.
- Canonical path checks prevent traversal, absolute-path use, directory delivery, and storage-root escapes.
- Evidence responses use private/no-store caching and safe attachment headers.
- Non-development requests cannot establish identity through legacy URL identity parameters.
- Stage and production keep the backend service behind the intended proxy boundary.
- The Rust runtime container builds with `Cargo.lock` and runs as a dedicated non-root user.
- Stage and production drop Linux capabilities and enable `no-new-privileges`.
- Startup output no longer exposes complete database URLs.

## Backup and recovery

- Production environment files are no longer copied into backup sets.
- Backup directories and files use restrictive permissions.
- Backup manifests and SHA-256 checksums are generated.
- Restore verifies checksums and archive structure before destructive operations.
- Restore requires explicit confirmation and stops application traffic before replacing data.

## Supply-chain and CI

- Cargo, Docker, and GitHub Actions dependencies are monitored by Dependabot.
- Rust builds and tests use the committed lockfile.
- CI validates Rust formatting, Clippy, tests, HTTP/DB smoke, Nix execution, Compose profiles, and the hardened Docker image.
- Rust advisory scanning and dependency license/source policy checks are part of the mandatory backend CI job.

## Maintainer and community readiness

- `main` is protected by pull-request, CI, linear-history, deletion, and force-push rules.
- `AGENTS.md` defines mandatory security invariants for Codex and other AI-assisted changes.
- `docs/THREAT_MODEL.md` documents assets, actors, trust boundaries, threats, controls, and known limitations.
- The primary README is fully English, with a separate maintained German overview.
- Pull requests #6 and #14 document the security changes and their validation history.

## Known limitations

- Evidence-download access decisions are emitted as structured runtime events; durable database-backed download-audit persistence remains a follow-up.
- An explicit lifecycle-state denial policy for withdrawn, rejected, expired, or retention-locked Evidence remains a follow-up.
- Backup manifests are checksummed but not cryptographically signed.
- Backup encryption depends on the operator's storage or transfer layer.
- Container bases and GitHub Actions are not yet all pinned by immutable digest or commit SHA.
- No independent penetration test has been completed.
